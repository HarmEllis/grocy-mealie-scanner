# Agent Instructions — grocy-mealie-scanner

## Language policy

All project artifacts MUST be written in English: source code, comments, log
strings, documentation (`README.md`, `AGENTS.md`, `BOARD_NOTES.md`, etc.),
commit messages, and any in-repo notes. Conversational chat with the
maintainer can be in Dutch, but the moment something lands in the repo it
switches to English. If you find existing Dutch content while editing,
translate it as part of your change.

## Devcontainer is mandatory

All firmware build steps and flash-server commands MUST run **inside the
devcontainer** (the VS Code devcontainer built from `.devcontainer/`).
Never run them directly on the host.

Find the running container for this workspace:

```bash
docker ps --filter "label=devcontainer.local_folder=$PWD" --format "{{.Names}}\t{{.Status}}"
```

Run commands inside the devcontainer — **always as the `node` user** via
`-u node`, and always source the ESP-IDF profile script:

```bash
docker exec -u node <container> bash -c "source /etc/profile.d/esp-idf.sh && <command>"
```

> **Important:** without `-u node` commands run as root and files (`build/`,
> `sdkconfig`, `dependencies.lock`) become root-owned, after which the
> `node` user can no longer build.

## Common commands

| Task | Command (inside the container) |
|------|--------------------------------|
| Build firmware | `cd /workspaces/grocy-mealie-scanner/firmware && idf.py build` |
| Set target (once after clean) | `cd /workspaces/grocy-mealie-scanner/firmware && idf.py set-target esp32s3` |
| Start flash server | `cd /workspaces/grocy-mealie-scanner/flash-server && bash serve.sh` |

## Project structure

```
firmware/          # ESP-IDF firmware (ESP32-S3, Freenove FNK0104AB 2.8")
  main/            # Application code
docs/              # DEVICE-API.md (contract with grocy-mealie-sync) + board notes
design/            # Claude design export the UI is built from (variant A)
flash-server/      # Web flasher (ESP Web Tools, port 8080)
  bins/            # Generated by serve.sh after idf.py build
```

The server side of the device API lives in the **grocy-mealie-sync** repo
(`../grocy-mealie-sync`, routes under `src/app/api/device/v1/`).
`docs/DEVICE-API.md` in this repo is the single source of truth for that
contract — change it here first, then implement on both sides.

## Hardware

See `firmware/BOARD_NOTES.md` for the full pin map (ILI9341 LCD, FT6336U
touch, GM67 wiring on UART1) and design-to-LVGL notes.

## idf_component.yml

Do **not** add `espressif/json` to `idf_component.yml` — `json` (cJSON) is a
built-in ESP-IDF component and is pulled in only via `CMakeLists.txt`
`REQUIRES json`.

## NVS writes

Keep NVS writes rare; the `nvs` partition must never move or grow
(partition-table changes do not propagate via OTA). NVS is for config that
must survive a cold boot: WiFi credentials, API base URL + device token, the
AP password. Per-scan state (last scanned product, UI state) never touches
NVS — use RTC memory or plain RAM.

## Release And Tag Prep

When asked to prepare or create a new release tag, complete all of the steps
below before creating the tag.

1. Determine the previous tag with `git describe --tags --abbrev=0` and review
   the changes since that tag with `git log --oneline <previous-tag>..HEAD`
   plus `git diff --stat <previous-tag>..HEAD`.
2. Update `CHANGELOG.md` for the upcoming release:
   - add the new version section at the top using `## [x.y.z] - YYYY-MM-DD`
   - summarize the changes since the previous tag in concise release notes
   - group entries under `Added`, `Changed`, and `Fixed` when that structure
     fits the changes
   - add or update the comparison link at the bottom so the new version
     compares `<previous-tag>...v<x.y.z>`
3. Bump the version in `flash-server/manifest.json` (the Pages workflow also
   rewrites this at release time, but keep the committed value in sync so
   local flashes report the right version).
4. Verify the release-prep edits before tagging. At minimum:
   - build the firmware: `cd firmware && idf.py build` inside the devcontainer
   - confirm `firmware/build/grocy-mealie-scanner.bin` is under the 3 MB OTA
     slot limit (`stat -c%s firmware/build/grocy-mealie-scanner.bin`)
   - confirm the partition table still contains `ota_0` and `ota_1`
     (`idf.py partition-table`)
5. Keep the release changes commit-ready first (changelog, version bump),
   but do not push the new `v<x.y.z>` tag yet.
6. As the final release step, ask the user whether to push the release.
   - only proceed when the user explicitly approves the push
   - when approved, push the release commit through the repository's allowed
     path to `main` (typically: push a release branch and merge a PR into
     `main`)
   - do not push the new `v<x.y.z>` tag until the release changes are merged
     into `main`
   - wait for the `CI` workflow on the merged `main` commit to complete
     successfully; if it fails, is cancelled, or never reaches a successful
     conclusion, stop and do not push the tag or create the GitHub release
   - before pushing the tag, create a GitHub draft release for `v<x.y.z>` on
     the merged `main` commit using the new changelog section as the release
     notes body, but omit the `## [x.y.z] - YYYY-MM-DD` heading from the body
     itself. The Pages workflow's `gh release create --generate-notes`
     fallback only fires when no release exists, so the curated draft must
     exist before the tag push to avoid being overwritten.
   - keep the short introductory summary paragraph from the changelog section
     at the top of the GitHub release notes body, before any `### Added`,
     `### Changed`, or `### Fixed` headings
   - always end the GitHub release notes with
     `**Full Changelog**: https://github.com/HarmEllis/grocy-mealie-scanner/compare/<previous-tag>...v<x.y.z>`
   - only after the curated draft release exists may the new `v<x.y.z>` tag
     be created on that merged `main` commit and pushed to the remote
   - after the tag push, the `Pages` workflow uploads the firmware binaries
     (`bootloader.bin`, `partition-table.bin`, `ota_data_initial.bin`,
     `grocy-mealie-scanner.bin`, `SHA256SUMS`) to the draft release and
     deploys the hosted Web Flasher
   - publish the draft release only after the workflow completes successfully
   - if the user does not approve, stop after the local release commit and
     report that the release has not been pushed, tagged remotely, or drafted
     on GitHub

## Version Format

- `flash-server/manifest.json` uses plain semver like `0.2.0`.
- Git tags use a `v` prefix like `v0.2.0`.

## Git safe.directory

The devcontainer runs under a different UID than the host. If git complains,
add this once:

```bash
docker exec <container> git config --global --add safe.directory /workspaces/grocy-mealie-scanner
```
